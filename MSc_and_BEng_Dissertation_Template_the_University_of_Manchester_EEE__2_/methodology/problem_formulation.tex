\subsection{Problem Formulation}

We formulate the UAV trajectory optimisation and resource allocation problem as a
maximisation of the total collected data throughput, subject to energy, kinematic, and
protocol constraints. The optimisation problem is defined as follows:
\begin{align}
    \text{(P1):} \quad \max_{\pi} \quad & \sum_{t=0}^{T} \sum_{i=1}^{N} D_{i,t}
        \label{eq:p1_obj} \\
    \text{s.t.} \quad
    & \text{RSSI}_{i,t}(p_t) \geq \text{RSSI}_{\min}(\text{SF}_{i,t}),
        \quad \forall i \in \mathcal{N} \label{eq:p1_c1} \\
    & E_{t+1} = E_t - P(a_t) \cdot \Delta t \geq 0,
        \quad \forall t \label{eq:p1_c2} \\
    & \sum_{i \in \mathcal{N}} \mathbb{I}(\text{SF}_{i,t} = k) \leq 1,
        \quad \forall k \in \{7, \ldots, 12\} \label{eq:p1_c3} \\
    & p_{t+1} = p_t + v(a_t) \cdot \Delta t \label{eq:p1_c4} \\
    & p_0 = p_{\text{start}},\ p_T = p_{\text{end}} \label{eq:p1_c5}
\end{align}
Where:
\begin{itemize}
    \item \(\mathcal{N}\): Set of \(N\) IoT sensor devices.
    \item \(\pi\): The control policy mapping states to actions \(a_t\).
    \item \(D_{i,t}\): Data collected from sensor \(i\) at time \(t\) (1 if successful, 0 otherwise).
    \item \(p_t\): UAV position \([x_t, y_t, z_t]^T\) at time \(t\).
    \item \(\text{RSSI}_{i,t}\): Signal strength received from sensor \(i\), calculated via the Two-Ray model.
    \item \(\text{SF}_{i,t}\): Spreading Factor assigned to sensor \(i\) by the ADR model.
    \item \(E_t\): Remaining UAV battery energy.
    \item \(\mathbb{I}(\cdot)\): Indicator function for collision detection (enforcing SF orthogonality).
\end{itemize}
This formulation captures the non-convex nature of the problem, where the decision variable
(UAV position \(p_t\)) non-linearly determines the link quality (RSSI) and protocol state
(SF), necessitating a Deep Reinforcement Learning approach.

\subsubsection{State Space Definition}

The state space \(\mathcal{S}\) is designed to provide the agent with effective awareness of
the network conditions and UAV state. At time step \(t\), the state vector
\(\mathbf{S}_t \in \mathcal{S}\) is defined as a tuple:
\begin{equation}
    \mathbf{S}_t = [p_t,\, E_t,\, B_t,\, U_t] \label{eq:state_vector}
\end{equation}
where:
\begin{itemize}
    \item \(p_t = [x_t, y_t]^T\): The 2D horizontal position of the UAV in the grid. The
    altitude is assumed constant at \(h_u\) metres, simplifying the control problem to
    horizontal trajectory optimisation. This design choice isolates the effects of path
    planning from altitude dynamics.

    \item \(E_t \in [0, E_{\max}]\): The remaining battery energy of the UAV (in joules or
    watt-hours). This constraint ensures the agent learns time-efficient trajectories within
    finite energy budgets.

    \item \(B_t \in [0, B_{\max}]^N\): A continuous vector representing the current data
    buffer occupancy level of each sensor \(i \in \mathcal{N}\). Rather than binary status
    (full/empty), we observe the actual buffer level in bytes, providing fine-grained
    information about imminent overflow risk. This allows the agent to make nuanced decisions
    about which sensors require urgent attention.

    \item \(U_t \in [0, 1]^N\): A normalised Urgency vector derived from each sensor's data
    loss rate and buffer utilisation. The urgency of sensor \(i\) is defined as:
    \begin{equation}
        U_{i,t} = \frac{B_{i,t}}{B_{\max}} \cdot \frac{\text{data\_generation\_rate}_i}{\text{collection\_rate}_i}
        \label{eq:urgency}
    \end{equation}
    This metric captures the immediate risk of buffer overflow: sensors with high buffer
    occupancy and fast data generation rates have high urgency. Unlike classical Age of
    Information (which only increases with time), urgency provides a denser reward signal
    tied directly to packet loss probability.
\end{itemize}

\paragraph{Dimensionality and Design Justification.}
The total dimensionality of the state space is \(3 + 2N\), where:
\begin{equation}
    \dim(\mathcal{S}) =
        \overbrace{2}^{\text{position}} +
        \overbrace{1}^{\text{battery}} +
        \overbrace{N}^{\text{buffers}} +
        \overbrace{N}^{\text{urgency}}
        = 3 + 2N
    \label{eq:state_dim}
\end{equation}

This represents a significant reduction from the theoretical maximum of \(5 + 3N\) (if we
included 3D position, binary buffer status, and spreading factors). Our design choices
prioritise computational efficiency while maintaining sufficient observability:

\begin{enumerate}
    \item \emph{2D vs.\ 3D Position}: We fix altitude to a constant \(h_u\), reducing
    positional dimensionality from 3 to 2. This is justified because the Two-Ray Ground
    Reflection model (\autoref{eq:tworay}) shows that height affects path loss through a
    quadratic term; thus, optimising vertical position would require balancing coverage
    against energy cost. For a BEng project, horizontal planning suffices to demonstrate
    protocol-aware navigation.

    \item \emph{Urgency as Proxy for AoI}: Rather than explicitly observing Age of
    Information~\cite{kaul2012realtime} (which would require tracking timestamp history), we
    use buffer occupancy and generation rate to infer urgency. This is physically motivated: a sensor with a nearly-full
    buffer is more urgent than a sensor with an empty buffer, regardless of how long data has
    been waiting. This provides immediate, actionable feedback for the agent.

    \item \emph{Spreading Factors Not Observed}: While
    \(\text{SF}_t \in \{7, \ldots, 12\}^N\) is not explicitly in the observation vector, the
    agent learns to infer spectral congestion through the dynamic changes in \(U_t\) and
    \(B_t\). When a sensor is stuck on SF12 (slow rate), its urgency will rise rapidly
    relative to nearby sensors. This implicit representation reduces dimensionality by an
    order of \(N\) while preserving the agent's ability to detect and avoid SF Monoculture.
\end{enumerate}

\paragraph{Frame Stacking for Temporal Context.}
To compensate for the removal of explicit SF observation, we employ frame stacking with
\(k = 4\) observations. This provides the agent with a temporal window of the last 4 steps,
allowing it to observe:
\begin{itemize}
    \item \emph{Urgency trends}: Is a sensor's urgency rising (approaching overflow) or stable?
    \item \emph{Collection effectiveness}: Are buffers decreasing after visits, or staying constant?
    \item \emph{Implied SF dynamics}: Rapidly rising urgency despite proximity suggests SF12 congestion.
\end{itemize}
The stacked state vector is:
\begin{equation}
    \mathbf{S}_t^{\text{stacked}} = [\mathbf{S}_{t-3},\, \mathbf{S}_{t-2},\, \mathbf{S}_{t-1},\, \mathbf{S}_t]
    \in \mathbb{R}^{4(3+2N)}
    \label{eq:stacked_state}
\end{equation}
This transformation makes the environment effectively Markovian despite being partially
observable (SF is not directly observed), enabling the DQN agent to infer temporal patterns
and make decisions based on network dynamics rather than instantaneous snapshots.

\paragraph{Summary.}
The state space design represents a principled trade-off between observability and
dimensionality. By using urgency as a proxy for AoI and inferring spectral state from buffer
dynamics, we achieve computational efficiency (\(3 + 2N\) dimensions per observation,
feasible for \(N = 20\) sensors), physical grounding (urgency directly measures packet loss
risk, not just staleness), and implicit protocol awareness (SF congestion is reflected in
urgency trends, detectable via frame stacking).

\subsubsection{Action Space Definition}

The action space \(\mathcal{A}\) is discrete to facilitate the use of the Deep Q-Network
(DQN) algorithm. At each time step, the agent selects one action \(a_t \in \mathcal{A}\):
\begin{equation}
    \mathcal{A} = \{\text{MoveN},\ \text{MoveS},\ \text{MoveE},\ \text{MoveW},\ \text{Hover}\}
    \label{eq:action_space_def}
\end{equation}
\begin{itemize}
    \item \emph{Movement Actions}: The four cardinal movement actions update the UAV's
    horizontal position by a fixed step size \(\delta_d\) (e.g., 1 metre). The altitude is
    assumed fixed in this study to isolate the trajectory optimisation problem.

    \item \emph{Hover Action}: The Hover action maintains the UAV's current position
    (\(p_{t+1} = p_t\)). This action is energetically expensive per unit distance (zero
    displacement) but is operationally essential. It allows the UAV to maintain a stable
    channel long enough to transmit large packets (high Time-on-Air) or to wait for a
    sensor's duty cycle window to open.
\end{itemize}
